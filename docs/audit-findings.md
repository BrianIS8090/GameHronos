# Хроно-Метро — Результаты аудита

Аудит запускается через `node js/test-audit.mjs`. Перебирает все 27 путей × все подмножества 7 улик (128 состояний = 3456 комбинаций) и проверяет, что текст концовки согласуется с выдаваемыми уликами.

> **Архитектура после рефакторинга 21.05.2026:** unlock-логика вынесена в чистую функцию `applyUnlocks(path, memoriesBefore, stateBefore)` в `js/rules.js`. Эта функция — единственный источник правды. `game.js` импортирует её и применяет результат через `unlockMemory`, тесты импортируют её же. Никаких больше копий, рассинхрона между runtime и тестами быть не может.

## Найденные дыры

### HIGH-1: `brake_broken` выдаётся за прыжок на террориста

**Путь:** `1-2-0` при `terrorist_know=true`.

**Что происходит:** Игрок выбирает «Сблизиться с парнем в капюшоне» → «Пытаться нейтрализовать» → «Прыгнуть на террориста и ударить по руке». Концовка: «ПРЯМАЯ АТАКА» (прыжок на террориста, никакого стоп-крана). Но `checkUnlocks` молча выдаёт `brake_broken` — улику про саботаж тормозов.

**Где:** `js/game.js:411`
```js
if ((i === 1 && j === 2 && k === 0) || ...) {
  this.unlockMemory('brake_broken');
}
```

**Фикс:** Условие должно требовать отсутствие `terrorist_know` (иначе путь означает другое действие):
```js
if ((i === 1 && j === 2 && k === 0 && !this.memories.terrorist_know) || ...)
```

При `terrorist_know=true` концовка `1-2-0` остаётся «ПРЯМАЯ АТАКА» — фатальная, без улики. Это согласовано: игрок физически не дёргает стоп-кран.

---

### HIGH-2: `yellow_wire` не выдаётся при стелс-нейтрализации террориста в сессии

**Путь:** `0-2-2` при `terroristNeutralized=true && soldier_trust=false`.

**Что происходит:** Игрок в текущем цикле прошёл стелс через `1-1-1` или `1-1-2` (Алексей сдерживает террориста, имя не назвал) и затем продолжил осмотр. На `CONTINUE` он выбрал ветку i=0 → бомба → жёлтый провод. Текст концовки — «ЖЕЛТАЯ НИТЬ» (потому что `terroristNeutralized` блокирует ветку «ПАРУСИНА И ОГОНЬ»). Но `checkUnlocks` требует именно `soldier_trust`, поэтому `yellow_wire` не записывается.

**Где:** `js/game.js:424`
```js
if (i === 0 && j === 2 && k === 2 && this.memories.soldier_trust) {
  this.unlockMemory('yellow_wire');
}
```

**Фикс:** Условие должно учитывать оба варианта нейтрализации террориста:
```js
if (i === 0 && j === 2 && k === 2 && (this.memories.soldier_trust || this.state.terroristNeutralized)) {
  this.unlockMemory('yellow_wire');
}
```

### HIGH-3 (UNLOCK-DESC ×3): Текст ПРЯМОЙ АТАКИ не упоминает тормоза

Это следствие HIGH-1: тот же путь `1-2-0` при `terrorist_know=true` выдаёт `brake_broken`, но в тексте нет ни одного упоминания тормозов. После фикса HIGH-1 эти 3 находки исчезнут автоматически.

### MEDIUM-1: `isFatalEnding` ложно классифицирует ИСТИННЫЙ РАССВЕТ как фатальную

**Что происходит:** В тексте ИСТИННОГО РАССВЕТА нет маркеров «ТИХИЙ СТЕЛС» или «Осмотр продолжается», поэтому `isFatalEnding` возвращает `true`. В игре это не проявляется, потому что `showEnding` сначала проверяет `isTrueFin` и подменяет кнопку. Но если использовать `isFatalEnding` где-то ещё (например, в будущих фичах) — будет дезориентация.

**Фикс:** Добавить в `isFatalEnding` явную проверку победы:
```js
export function isFatalEnding(i, j, k, memories, state = {}) {
  const text = getEndingText(i, j, k, memories, state);
  if (text.includes('ИСТИННЫЙ РАССВЕТ')) return false;
  return !text.includes('ТИХИЙ СТЕЛС') && !text.includes('Осмотр продолжается');
}
```

## Дополнительные нюансы (не баги, но требуют внимания)

### N-1: vA не покрывает 0-2-x при `bomb_found`

Когда у игрока есть `bomb_found`, на путях `0-2-0`, `0-2-1`, `0-2-2` концовки берутся из `staticEndings` — общих текстов, где игрок «судорожно вскрывает рюкзак». По смыслу, с уликой `bomb_found` игрок уже знал про бомбу и подходит к ней целенаправленно, поэтому тексты должны звучать иначе. Этот пункт — улучшение, а не дыра.

### N-2: vC не покрывает 2-1-2 при `generator_key`

Аналогично: `2-1-2` с ключом даёт «ВОЗДУШНЫЙ СДВИГ» из `staticEndings`. Текст работает с обоими состояниями, но эмоционально игрок «открывает люк ключом» — с `generator_key` это естественно, без него — обманчиво.

### N-3: cabin_code и подсветка кнопок keypad

Сейчас сломанные кнопки keypad (6 из 9) подсвечиваются классом `.broken` всегда, вне зависимости от наличия `cabin_code`. По задумке `cabin_code` должна была раскрывать уязвимость. Игрок без улики всё равно видит подсветку, что обесценивает её ценность. Решается либо скрытием подсветки без улики, либо явной защитой в `showEnding` (`СЛУЧАЙНЫЙ ВЗЛОМ` уже стоит — даже с угаданным кодом без `cabin_code` побед не будет).

Оставляем как есть: подсветка нужна для UX, а провал в финале защищает от чита.

### N-4: narrative-map.md устарел

- В таблице путей `yellow_wire` указан как безусловная разблокировка `0-2-2`. На деле требуется `soldier_trust` (или после фикса HIGH-2 — `soldier_trust || terroristNeutralized`).
- В разделе «Оптимальный порядок сбора улик» указано 9 циклов. Минимум — 8 (см. `state-graph.md` пункт 8).
- Цикл «1-2-2 (1) → имя Алексея» и «1-2-2 (2) → soldier_trust» — на самом деле это один цикл: `soldier_trust` устанавливается в первый же раз через `1-2-2 + terrorist_know`.

## Сценарии, которые теперь покрыты тестами

`js/test-audit.mjs` запускает 19 проверок:

1. **BB** — Достижимость веток после `terrorist_know` и `terroristNeutralized`
2. **UNLOCK-DESC** — Согласованность текста концовки с выдаваемой уликой (×7 типов)
3. **REACH** — Каждая улика достижима хотя бы одним путём из чистого старта
4. **OPT** — Симуляция оптимального прохождения 7+1 циклов до победы
5. **TYPO** — Известные опечатки
6. **YW-MISS** — `yellow_wire` после стелс-нейтрализации в сессии
7. **BRK-MISFIRE** — `brake_broken` не выдаётся через прыжок на террориста
8. **DISC** — Первая встреча с террористом через ветку i=1
9. **FAIL-WRONG** — 6 отдельных provala 2-2-2 с правильными подсказками
10. **HTML-IMBALANCE** — Балансировка `<strong>` и `<em>` во всех концовках
11. **NO-RESONANCE** — Каждой улике соответствует хроно-резонанс в сцене 1
12. **STEALTH-NO-YW** — Сценарий 1-1-1 (стелс) → CONTINUE → 0-2-2 → `yellow_wire`
13. **WIN-LEAK** — Победа не выпадает при недостающих уликах
14. **HTML-NO-STRONG** — У каждой концовки есть заголовок
15. **ALT-UNREACH** — Все альтернативные пути к уликам действительно работают
16. **CHAIN-NO-BB** — `1-2-0` из чистого состояния даёт `terrorist_know` + `brake_broken` одновременно (regression по HIGH-1)
17. **YW-LEAK / YW-BLOCK** — `yellow_wire` требует `terrorist_know && soldier_trust` или `terroristNeutralized` (regression по HIGH-2)
18. **NO-CASCADE** — Любой выбор на ветке 0-2 (вскрытие рюкзака) каскадно даёт `bomb_found`
19. **TX-NOMATCH 1-2-0** — Текст и улика `brake_broken` на старой ветке согласованы

Запуск: `node js/test-runner.mjs` (включает оба тестера).
