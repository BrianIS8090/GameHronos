# Технический аудит интерактивной истории «Хроно-Метро»

Дата проверки: 21.05.2026  
Архив: `интерактивная история.zip`  
Проверенные основные файлы: `js/game.js`, `js/text-data.js`, `js/test-audit.mjs`, `js/test-runner.mjs`, `docs/*.md`

---

## 1. Краткое резюме

Игра **проходима до истинной концовки**: все 27 базовых путей имеют текст, финал `2-2-2` достижим, а комплектные тесты `node js/test-runner.mjs` проходят с результатом `Critical: 0, High: 0, Medium: 0, Low: 0`.

Однако при ручной и дополнительной автоматической проверке фактической логики найдены проблемы, которые комплектный аудит не ловит. Главные риски:

1. **После отмены keypad-мини-игры можно получить `2-2-2` / истинную концовку без повторного решения кода.** Также после отмены/провала keypad таймер не возобновляется.
2. **Путь `1-2-0` из чистого состояния показывает игроку саботаж тормозов, но фактически не записывает `brake_broken`** из-за порядка мутации `this.memories` в `checkUnlocks`.
3. **`0-2-2` может выдать `yellow_wire`, когда игрок знает только имя Алексея, но не знает террориста.** В тексте Алексей внезапно удерживает террориста, хотя персонаж ещё не установлен.
4. **`yellow_wire` можно получить без `bomb_found`, после чего финальная подсказка `2-2-2` может утверждать, что игрок “даже не подозревает” о бомбе, хотя он уже знает жёлтый провод.**
5. **Ветка тормозного люка `2-1-x` логически противоречит себе:** два варианта говорят, что шланги исправны, а третий внезапно предлагает зажать “повреждённый шланг”.

Приоритет исправления: сначала keypad и `checkUnlocks`, затем условия `0-2-2`, затем текстовые/UX-несостыковки.

## 2. Методика проверки

Проверка проводилась в четыре слоя:

1. **Инвентаризация исходников:** структура архива, игровые модули, существующая документация и автотесты.
2. **Запуск комплектных тестов:** `node js/test-runner.mjs`.
3. **Ручная трассировка всех 27 путей:** выборы `i-j-k`, условные варианты по `memories` и `state.terroristNeutralized`, тип исхода `RESTART / CONTINUE / TRUE FIN`.
4. **Дополнительная проверка фактической мутации состояния:** отдельная симуляция порядка `checkUnlocks` именно как в `js/game.js`, а не как в тестовой копии.

Комплектный тест проходит, но в `js/test-audit.mjs` используется копия `checkUnlocks`, которая отличается от поведения реального `game.js` в важном месте. Из-за этого часть проблем скрыта.

## 5. Найденные проблемы

## CRITICAL-1. Keypad cancel/back ломает путь и позволяет обойти мини-игру

**Файлы:** `js/game.js:284-302`, `js/game.js:310-315`, `js/game.js:446-447`, `js/game.js:580-586`, `js/game.js:672-691`

### Что происходит

В `handleChoice` выбор `2-2-2` добавляется в `pathHistory` до запуска keypad:

```js
this.state.pathHistory.push(choiceIndex);
...
if (i === 2 && j === 2 && choiceIndex === 2) {
  this.startKeypadGame();
}
```

При отмене keypad:

```js
_keypadCancel() {
  this.keypadState = null;
  const overlay = document.getElementById('keypad-overlay');
  if (overlay) overlay.remove();
  this.state.isGameOver = false;
  this.audio.playGlitch();
  this.renderScene();
}
```

`pathHistory` не очищается. Внутри остаётся `[2, 2, 2]`.

Дальше игрок возвращается на сцену 3 и выбирает, например, `k=0`. Новый выбор добавляется четвёртым элементом: `[2, 2, 2, 0]`. Но `showEnding()` берёт `k` так:

```js
const k = this.state.pathHistory[2] !== undefined ? this.state.pathHistory[2] : 0;
```

То есть игра всё равно считает путь как `2-2-2`, хотя игрок после отмены нажал другой вариант.

### Воспроизведение

1. Дойти до `2-2-2`.
2. Открыть keypad.
3. Нажать `ОТМЕНА`.
4. На сцене 3 выбрать `k=0` или `k=1`.
5. Игра покажет не `2-2-0` / `2-2-1`, а `2-2-2`.
6. Если все 7 улик уже собраны, это потенциально даёт истинную концовку без решения keypad.

### Дополнительный эффект

`startKeypadGame()` останавливает таймер:

```js
if (this.timerInterval) clearInterval(this.timerInterval);
```

После `_keypadCancel()` таймер не запускается заново. После `_keypadFail()` при возврате кнопкой `BACK` таймер также не возобновляется. Это позволяет заморозить время на сцене 3.

### Ожидаемое поведение

- При отмене keypad путь должен откатываться с `[2,2,2]` до `[2,2]`.
- После отмены/возврата от провала keypad таймер должен продолжать идти с текущего `timeRemaining`.
- Истинный финал должен быть доступен только после успешного keypad-submit.

### Рекомендованный фикс

Минимальный фикс:

```js
_keypadCancel() {
  this.keypadState = null;
  const overlay = document.getElementById('keypad-overlay');
  if (overlay) overlay.remove();

  if (this.state.pathHistory.length === 3 &&
      this.state.pathHistory[0] === 2 &&
      this.state.pathHistory[1] === 2 &&
      this.state.pathHistory[2] === 2) {
    this.state.pathHistory.pop();
  }

  this.state.currentScene = 3;
  this.state.isGameOver = false;
  this._resumeTimerFromCurrentRemaining();
  this.audio.playGlitch();
  this.renderScene();
}
```

Также нужно добавить общий метод `_resumeTimerFromCurrentRemaining()`, чтобы не копировать логику `setInterval` в нескольких местах.

Для `_keypadFail()` в обработчике `BACK` также нужно вызвать откат пути и возобновление таймера.

## HIGH-1. `1-2-0` из чистого состояния не выдаёт `brake_broken`, хотя текст говорит о саботаже тормозов

**Файлы:** `js/game.js:403-416`, `js/test-audit.mjs:8-25`, `js/text-data.js:427`

### Что происходит

В тексте базового пути `1-2-0` игрок узнаёт сразу две вещи: стоп-кран бесполезен из-за саботажа тормозов, а парень в капюшоне — террорист.

Текст концовки:

```text
КОНЦОВКА №16: СОРВАННЫЙ ТОРМОЗ (САБОТАЖ + ТЕРРОРИСТ ОБНАРУЖЕН!)
...
[Стоп-кран бесполезен — тормоза саботированы! Парень в капюшоне — настоящая угроза!]
```

Но реальный `checkUnlocks` сначала записывает `terrorist_know`:

```js
if ((i === 1 && j === 0 && k === 2) || (i === 1 && j === 2 && k === 0)) {
  this.unlockMemory('terrorist_know');
}
```

А затем проверяет `brake_broken` через уже изменённое `this.memories.terrorist_know`:

```js
if ((i === 1 && j === 2 && k === 0 && !this.memories.terrorist_know) ||
    (i === 2 && j === 1 && k === 2 && this.memories.generator_key)) {
  this.unlockMemory('brake_broken');
}
```

Из-за этого из чистого состояния путь `1-2-0` фактически выдаёт только:

```text
terrorist_know
```

А должен выдавать:

```text
terrorist_know + brake_broken
```

### Почему комплектный аудит это не ловит

В `js/test-audit.mjs` копия `checkUnlocks` проверяет `!memories.terrorist_know`, то есть старое состояние до мутации:

```js
if ((i === 1 && j === 2 && k === 0 && !memories.terrorist_know) || ...)
```

В реальном `game.js` проверяется уже мутированное `this.memories.terrorist_know`. Поэтому тестовая реплика оптимистичнее реальной игры.

### Влияние на игру

- Игрок получает текст “саботаж обнаружен”, но архив `brake_broken` не открывается.
- Игрок может позже прийти на финал и получить провал по саботажу тормозов, хотя уже видел соответствующую концовку.
- Документация и тесты могут утверждать, что путь работает, хотя в runtime он не работает.

### Рекомендованный фикс

Использовать снимок памяти до любых unlock-мутаций:

```js
checkUnlocks(i, j, k) {
  const hadTerroristKnow = this.memories.terrorist_know;
  const hadGeneratorKey = this.memories.generator_key;
  const hadSoldierTrust = this.memories.soldier_trust;

  if (i === 0 && j === 1 && (k === 1 || k === 2)) {
    this.unlockMemory('bomb_found');
  }

  if ((i === 1 && j === 0 && k === 2) || (i === 1 && j === 2 && k === 0)) {
    this.unlockMemory('terrorist_know');
  }

  if (i === 2 && j === 1 && (k === 0 || k === 1)) {
    this.unlockMemory('generator_key');
  }

  if ((i === 1 && j === 2 && k === 0 && !hadTerroristKnow) ||
      (i === 2 && j === 1 && k === 2 && hadGeneratorKey)) {
    this.unlockMemory('brake_broken');
  }

  // остальные условия аналогично лучше считать от pre-state,
  // если смысл ветки зависит от того, что игрок знал ДО выбора.
}
```

Лучшее архитектурное решение: вынести unlock-логику в чистую функцию `applyUnlocks(path, memoriesBefore, stateBefore)`, экспортировать её и использовать и в игре, и в тестах. Тогда тесты не будут расходиться с runtime.

## HIGH-2. `yellow_wire` выдаётся при одном `soldier_trust`, даже если террорист ещё не установлен

**Файлы:** `js/game.js:431-433`, `js/text-data.js:347-355`, `js/game.js:423-425`

### Что происходит

Сейчас `yellow_wire` выдаётся так:

```js
if (i === 0 && j === 2 && k === 2 &&
    (this.memories.soldier_trust || this.state.terroristNeutralized)) {
  this.unlockMemory('yellow_wire');
}
```

Это означает, что достаточно знать имя/доверие Алексея. Знать, кто террорист, необязательно.

Состояние воспроизводимо:

1. Из чистого состояния пройти `1-2-1`.
2. Получить `soldier_trust`, но не получить `terrorist_know`.
3. Пройти `0-2-2`.
4. Игра покажет `ЖЁЛТАЯ НИТЬ` и выдаст `yellow_wire`.

Проблема в тексте: Алексей внезапно удерживает террориста, хотя игрок ещё не знает, кого именно нужно нейтрализовать.

Фрагмент логики `0-2-2`:

```js
if (memories.soldier_trust) {
  return "... В дальнем углу майор Алексей ... пульт уже вылетел из руки террориста ...";
}
```

### Почему это ломает логику

`trust` к Алексею не равен знанию цели. Алексей может доверять игроку, но игрок без `terrorist_know` не должен иметь возможность указать ему на конкретного террориста. Сейчас событие “Алексей удерживает террориста” появляется из ниоткуда.

### Ожидаемое поведение

Для безопасного обезвреживания бомбы должны быть выполнены одно из условий:

- `terrorist_know && soldier_trust`: игрок знает террориста и может доверенно направить Алексея;
- `state.terroristNeutralized`: террорист уже обезврежен в текущей сессии через stealth-сцену `1-1-1` или `1-1-2` при `terrorist_know=true`.

### Рекомендованный фикс

В `checkUnlocks`:

```js
const terroristControlled =
  (this.memories.terrorist_know && this.memories.soldier_trust) ||
  this.state.terroristNeutralized;

if (i === 0 && j === 2 && k === 2 && terroristControlled) {
  this.unlockMemory('yellow_wire');
}
```

В `getEndingText(0,2,2,...)`:

```js
const terroristControlled =
  (memories.terrorist_know && memories.soldier_trust) ||
  state.terroristNeutralized;

if (!terroristControlled) {
  return "ПАРУСИНА И ОГОНЬ"; // или отдельный fail-вариант
}
```

Желательно сделать отдельную fail-концовку для состояния `soldier_trust=true && terrorist_know=false`: игрок знает Алексея, но не знает, кого именно нейтрализовать.

## MEDIUM-1. `yellow_wire` можно получить без `bomb_found`, а финальный fail-текст затем противоречит памяти игрока

**Файлы:** `js/game.js:399-400`, `js/game.js:431-433`, `js/text-data.js:322-323`, `js/text-data.js:347-355`

### Что происходит

`bomb_found` выдаётся только на `0-1-1` и `0-1-2`:

```js
if (i === 0 && j === 1 && (k === 1 || k === 2)) {
  this.unlockMemory('bomb_found');
}
```

Но `yellow_wire` можно получить на `0-2-2`, где игрок буквально вскрывает рюкзак, видит провода и перекусывает жёлтый. При этом `bomb_found` не записывается.

Возможное состояние:

```text
yellow_wire=true
bomb_found=false
```

После этого финальная проверка `2-2-2` первым условием делает так:

```js
if (!(memories.bomb_found && memories.yellow_wire)) {
  return "ТАЙНА МЕДНЫХ ЖИЛ";
}
```

Текст говорит:

```text
... под сиденьем №4 в салоне тикает бомба, а я даже не подозреваю о ней.
```

Это противоречит состоянию `yellow_wire=true`: игрок уже знает внутренности рюкзака и безопасный провод.

### Влияние

- Игрок может иметь более глубокую улику (`yellow_wire`), но не иметь базовую (`bomb_found`).
- Финальная подсказка становится неверной для части состояний.
- Система памяти выглядит нелогично: знание жёлтого провода не подразумевает знание факта бомбы.

### Рекомендованные варианты фикса

Вариант A — каскадное открытие базовой улики:

```js
if (i === 0 && j === 2 && k === 2 && terroristControlled) {
  this.unlockMemory('bomb_found');
  this.unlockMemory('yellow_wire');
}
```

Можно также открыть `bomb_found` на `0-2-0` и `0-2-1`, потому что игрок тоже видит рюкзак и провода.

Вариант B — разделить финальную проверку:

```js
if (!memories.bomb_found) {
  return "... я не знаю, где бомба ...";
}
if (!memories.yellow_wire) {
  return "... я знаю про бомбу, но не знаю безопасный провод ...";
}
```

Оптимально совместить оба варианта: базовая улика должна открываться при любом осмысленном вскрытии рюкзака, а финальная подсказка должна точно объяснять, чего именно не хватает.

## MEDIUM-2. Ветка тормозного люка `2-1-x` противоречит себе

**Файлы:** `js/text-data.js:180-183`, `js/text-data.js:297-299`, `js/text-data.js:400-401`, `js/text-data.js:435`

### Что происходит

При `generator_key=true` сцена `2-1` открывает инспекционный люк и показывает тормозные шланги.

Выборы:

```js
{ text: "Аккуратно осмотреть тормозную магистраль в люке" }
{ text: "Последовательно проверить каждый шланг" }
{ text: "Попробовать зажать поврежденный шланг руками" }
```

Проблемы:

1. Третий выбор уже называет шланг “повреждённым”, хотя до этого сцена не сообщала о повреждении.
2. `2-1-0` говорит: “Всё выглядит исправным — никаких видимых повреждений”.
3. `2-1-1` говорит: “Все соединения плотные, утечек не видно”.
4. `2-1-2` говорит: “вижу перебитый шланг”.

Это создаёт противоречие: методичный осмотр не видит повреждения, а попытка зажать шланг внезапно его обнаруживает.

### Влияние

- Игроку сложно понять, почему правильный путь — наиболее рискованное действие.
- Если `brake_broken` уже открыта, повторный выбор `2-1-0` / `2-1-1` всё равно говорит, что шланги исправны, что прямо спорит с хроно-резонансом.

### Рекомендованный фикс текста

Сцена `2-1` при `generator_key=true` должна заранее дать слабый сигнал:

```text
В глубине люка слышен едва заметный свист воздуха. Среди переплетения шлангов один участок уходит в тень; без внимательной проверки не понять, где утечка.
```

Выбор `k=2` лучше заменить на нейтральный:

```text
Нащупать источник утечки и попытаться временно перекрыть магистраль
```

Тексты `2-1-0` / `2-1-1` лучше сделать не “всё исправно”, а “осмотр не дал результата”:

```text
Снаружи повреждение не видно: шланги уходят глубже под пол, а свист теряется в грохоте колёс. Я закрываю люк, понимая, что нужен более рискованный способ проверки.
```

Если `brake_broken=true`, эти варианты должны признавать уже известный факт:

```text
Я уже знаю, что один из шлангов надрезан, но с этой позиции не могу безопасно перекрыть утечку.
```

## MEDIUM-3. Комплектный аудит не является source-of-truth для unlock-логики

**Файлы:** `js/test-audit.mjs:6-25`, `js/game.js:398-435`

### Что происходит

`test-audit.mjs` содержит ручную копию `checkUnlocks`. Эта копия уже расходится с runtime-поведением на `1-2-0`.

Фактическая дополнительная проверка всех подмножеств памяти показала расхождение на `1-2-0` для 32 pre-state состояний, где `terrorist_know=false`: тестовая копия ожидает `brake_broken`, реальный порядок `game.js` его не выдаёт.

### Риск

Будущие правки могут снова “починить тест”, но не игру, или наоборот. Для нарративной логики это особенно опасно, потому что тесты проверяют не DOM, а отдельную модель.

### Рекомендованный фикс

Создать отдельный модуль, например `js/rules.js`:

```js
export function applyUnlocks(path, memoriesBefore, stateBefore) {
  const next = { ...memoriesBefore };
  const fired = [];
  const unlock = (key) => {
    if (!next[key]) {
      next[key] = true;
      fired.push(key);
    }
  };

  // вся логика unlock здесь
  return { memories: next, fired };
}
```

Далее:

- `game.js` использует `applyUnlocks` и применяет `fired` через `unlockMemory` для UI-баннеров;
- `test-audit.mjs` импортирует ту же функцию;
- тесты больше не дублируют игровую логику.

## LOW-2. Рандомный glitch в typewriter может ломать HTML-разметку

**Файл:** `js/game.js:244-268`

### Что происходит

`typewriterEffect` разбивает HTML-текст на слова и иногда оборачивает слово в:

```js
<span class="glitch-text">...</span>
```

Если рандом попадает на слово с HTML-тегом, например `<strong>КОНЦОВКА`, восстановление идёт через `textContent`, из-за чего тег может стать видимым текстом или форматирование может временно ломаться.

### Рекомендованный фикс

Не применять glitch к токенам, содержащим `<` или `>`:

```js
if (!word.includes('<') && !word.includes('>') && Math.random() < 0.04 && word.length > 3) {
  // glitch
}
```

Более надёжно — рендерить текст через безопасный токенизатор HTML, а glitch применять только к текстовым нодам.

## 8. Рекомендации по улучшению игрового опыта

1. **Сделать финальные подсказки более точными.** Сейчас первый fail `2-2-2` объединяет “не знаю бомбу” и “не знаю провод”. Лучше разделить эти случаи.
2. **Явно показать связь `terrorist_know → soldier_trust → yellow_wire`.** Игрок должен понимать, что Алексей полезен только когда ему можно указать цель.
3. **Переписать ветку тормозного люка как диагностическую мини-ветку.** Сейчас правильный выбор выглядит случайным. Лучше дать слабую улику: свист, запах, вибрация, тень/надрез на шланге.
4. **После открытия улики менять повторные тексты с “вы открыли” на “вы используете”.** Это уменьшит ощущение, что игра повторно выдаёт уже известную информацию.
5. **Отделить “знание объекта” от “знания решения”.** Например: `bomb_found` — где бомба; `yellow_wire` — как обезвредить. Тогда любые ветки, где игрок видит рюкзак, должны как минимум открывать `bomb_found`.
6. **Сделать тесты частью игровой модели, а не документации.** Нарративные правила лучше хранить в коде как чистые функции и тестировать их напрямую.
